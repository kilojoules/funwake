\section{Discussion}
\label{sec:discussion}

\subsection{The Schedule Interface as a Search Space}

The schedule-only interface constrains the LLM to a 4-dimensional
function space---but this space is surprisingly rich. The two winning
schedules use qualitatively different mechanisms (Gaussian bumps vs.\
sinusoidal shake) to solve the same problem (escaping local optima
while maintaining feasibility). This suggests the search space is not
saturated: there may be other undiscovered schedule structures that
further improve generalization.

The benchmark's value lies in this gap between interface simplicity
and solution richness. The function signature is trivial to implement
(4 inputs, 4 outputs, pure Python), evaluation is cheap (30--60
seconds, CPU only), and the held-out physics prevents overfitting.
But the schedules that maximize held-out performance require
non-obvious reasoning about optimization dynamics---specifically,
about the interaction between learning rate decay and constraint
penalty ramping.

\subsection{Why the Held-Out Farm Matters}

Two specific failure modes are invisible without held-out evaluation:

\paragraph{Penalty coupling failure.}
Schedules that reduce $\alpha$ at convergence produce feasible layouts
on the spacious training polygon (DEI, 50 turbines, 960\,m spacing)
but infeasible layouts on the tighter held-out polygon (ROWP, 74
turbines, 792\,m spacing). The training farm's generous geometry masks
weak constraint handling---AEP remains high even with mild boundary
violations, because turbines near the boundary still contribute power.
On the held-out farm, violations push turbines outside the polygon
entirely.

\paragraph{Training-optimal $\neq$ held-out-optimal.}
Gemini's training-best schedule (AEP = 5600.0) achieves ROWP = 4255.1.
Its held-out-best schedule (ROWP = 4328.0) achieves only training
AEP = 5552.1. The schedules that maximize training AEP do so partly
by under-enforcing constraints---gaining a few GWh from near-boundary
placement that happens to be feasible on DEI but not ROWP. This
divergence cannot be detected without a physically distinct held-out
problem.

\subsection{Benchmark Properties}

\paragraph{Cost.}
Evaluating a submitted schedule requires only a CPU and takes
30--60 seconds. Running an LLM agent through the benchmark costs
\$5--50 for a 5-hour session depending on model. The 500-multi-start
baseline costs $\sim$2.5 GPU-hours per farm. Full paper reproduction
costs $\sim$\$500. This is 5--10$\times$ cheaper than SWE-bench
evaluations at comparable scope.

\paragraph{Difficulty scaling.}
The DEI farm variants ($N = 30, 40, 50, 60, 70, 80, 100$) provide
a natural difficulty axis. Low $N$ is easy (sparse turbines, weak
wake interactions); high $N$ is hard (tight packing, strong
constraint pressure). The ROWP held-out farm adds a cross-physics
axis (different turbine, polygon, wind).

\paragraph{Not saturated.}
The spread between the best (Gemini, +81.3~GWh on ROWP) and worst
successful agent (Llama, +5.3~GWh) spans an order of magnitude.
The two winning schedules are qualitatively different, suggesting
multiple unexplored regions of the schedule space. We expect stronger
models and longer time budgets to find further improvements.

\subsection{Limitations}

\paragraph{Single training farm.}
The LLM trains on one farm (DEI) and generalizes to variants of it
plus one distinct held-out farm (ROWP). Additional training scenarios
would test robustness to training distribution.

\paragraph{Schedule-only mode fixes initialization.}
The skeleton uses wind-aware grid initialization. In full-optimizer
mode, agents improve initialization (hexagonal lattice, K-means
clustering). The schedule-only interface cannot express initialization
innovations, which limits what it measures.

\paragraph{No random search ablation.}
We have not yet compared LLM-generated schedules against random or
grid search over parameterized schedule families with the same
evaluation budget. This ablation would isolate the LLM's contribution
from the schedule parameterization itself.

\paragraph{Agent scaffold differences.}
Frontier models (Claude, Gemini) use rich CLI agents; open-source
models use structured-output prompting with injected context. While
we ensure equivalent information access, the agent frameworks differ
in their ability to iteratively debug failing schedules.
